摩擦指数

评估方法论 · 开发者旅程摩擦指数

本报告的所有分数,均由可观测、可计数的指标经公开公式计算得出,不含逐项主观打分。框架名为 摩擦指数(Developer Journey Friction Index)——测量开发者完成真实任务时遇到的、可被复现测量的摩擦。给定测量值与公式,任何人重算得同一结果。

一、理论依据

ISO 9241-11:2018
可用性的国际标准定义——有效性 / 效率 / 满意度三要素,构成评估维度的总框架。
Nielsen / NN/g
启发式评估、交互成本(Interaction Cost)与竞品可用性基准测试法——竞品同维度对比的标准做法。
DevEx Framework(CACM 2023)
Noda、Forsgren 等提出的开发者体验框架——反馈循环 / 认知负荷 / 心流,为指标提供"开发者专属"颗粒度。
认知负荷与交互定律
Sweller 认知负荷理论、Hick 定律(选项越多决策越慢)、Fitts 定律、Tesler 复杂度守恒定律。
Time-to-First-Run
开发者关系(DevRel)通用黄金指标——从落地到第一次成功运行的成本,衡量上手速度。
Quiñones & Rusu(2017)
《如何构建可用性启发式》系统综述——本框架领域指标的构建方法论依据。
二、评估对象与边界(精确界定,不越界)
测量:开发者门户中可观测、可计数的旅程摩擦——点击成本、准入门槛、决策负荷、内容可执行性、可发现性。
不纳入计算:视觉美感、文案质量、主观"清晰度"感受、品牌好感——这些无法在不引入判断的前提下量化,故不计入分数,仅作不计分的"定性观察"另列。
适用范围:PC 端 Web 开发者门户;移动 SDK / IDE 插件 / 端侧部署需各自扩展指标层。

三、指标体系(5 维度 · 全部可观测)

A · 效率与反馈循环
到达目标点击/跳转数 · 完成总步骤数 · 上下文切换数 · Time-to-First-Run。依据:ISO 效率 / NN/g 交互成本 / DevEx
B · 认知负荷
版本/平台选择项数 · 必做决策点数 · 目标导航深度(IA 层级)。依据:Sweller / Hick 定律 / DevEx
C · 准入摩擦
是否需登录/注册(0/1) · 是否需同意协议/审批(0/1) · 必填表单字段数。依据:Nielsen 用户控制 / Tesler 定律
D · 内容可执行性
代码示例可否复制即运行(0/1) · 快速入门有无完整可运行代码(0/1) · 文档时效。依据:Nielsen 帮助与文档 / DevEx
E · 可发现性
核心关键词搜索命中率 · 路径死链/404 数 · 关键入口可达性。依据:Nielsen 识别而非回忆
四、执行 · 测量协议(保证可复现)
① 设备/视口:桌面 1440×900,Chrome 最新稳定版;② 账号状态:未登录态进入(模拟首次开发者);③ 计数规则:点击 = 一次有效交互,步骤 = 用户必须完成的离散动作,同一动作不重复计;④ 取证:每个指标值附对应截图为证;⑤ 时点:记录评估日期与页面版本;⑥ 缺测处理:无法实测的指标标注"缺测"并排除出计算,不以估值填充。
五、执行 · 归一化与计算(全公式,无逐项判断)
归一化(测量值 → 0–100,分越高 = 摩擦越小):
・数值型:分 = ( worst − 实测值 ) ÷ ( worst − best ) × 100,截断到 0–100;
・布尔型(0/1):无门槛 = 100,有门槛 = 0;　・比例型:直接 ×100。
锚点 best/worst 取理论最优与合理劣化上限,全部公开、对两边一视同仁。
合成:维度分 = 维度内指标分算术平均;触点分 = 5 维度分算术平均(等权,去除权重主观);步骤分 / 总分 = 触点分算术平均。
敏感性分析:另给"按 DevEx 维度加权"的备选结果,验证主结论在不同权重下是否稳定。

七、可复现性、效度与局限(诚实陈述)
可复现性(强):测量协议 + 公开锚点 + 公开公式 ⇒ 给定输入,输出唯一。
构念效度(中):指标均挂靠权威构念,但"可观测摩擦"是体验的代理,非体验全部(见边界)。
